PromptOps — lifecycle of a prompt

You type “implement feature X.” Here's everywhere that goes.

PromptOps doesn't change what Claude gives you in the moment — it records everything about the attempt, scores it against what actually happened in your codebase, and uses that history to recommend a better starting point next time. Below is that path, stage by stage, with what's actually running today versus what's designed but not yet built.

Live now Designed, not built yet PromptVersion artifact this stage produces
  1. 01

    Pick — or get recommended — a prompt

    live

    Actor: you, or /promptops recommend via IActivityClassifier + IRecommendationProvider

    You write “implement feature X” as a new PromptVersion — or paste it as a task description via /promptops recommend and let the daemon classify it into activity tags and rank your best-performing prior versions for tasks like it, blending semantic similarity, tag overlap, and historical score. First time here? /promptops init seeds a starter catalog covering common activities (bug fixes, tests, refactors, reviews, and more) so there's something to recommend from day one.

    produces Prompt PromptVersion PromptMetadata
  2. 02

    Claude Code session starts

    live

    Actor: SessionStart hook (thin, per-repo plugin)

    The hook reads your repo locally — it's the only thing that can; the daemon has no filesystem access — and opens an execution: repository, branch, commit, detected language, the task. Nothing about your code leaves the loopback connection to your own machine.

    produces ExecutionRecord · InProgress
  3. 03

    Claude actually does the work

    untouched by PromptOps

    Actor: Claude Code, exactly as always

    This step is untouched. Claude reads files, edits code, runs commands exactly as it would without any of this installed. Hooks quietly log which tools ran, how often, and for how long.

    produces ExecutionToolUsage × N
  4. 04

    Session ends

    live

    Actor: Stop hook

    The hook computes what actually changed — files touched, lines added and removed — and closes out the execution with Claude's output. This is the moment a full record exists, and it's what the rest of the pipeline reacts to.

    produces ExecutionRecord · Finished event: ExecutionRecorded
  5. 05

    Engineering signal arrives

    live

    Actor: IMetricCollector plugins (Sonar, build/test)

    Once CI runs — maybe seconds later, maybe after you push — build results, test outcomes, and static-analysis findings get attached back to this same execution.

    produces EngineeringMetrics
  6. 06

    You rate it — optional

    live

    Actor: you, via /promptops rate

    Correctness, helpfulness, whether it introduced complexity you didn't ask for — your judgment, captured as structured data instead of a fading impression.

    produces HumanEvaluation
  7. 07

    The model judges it too

    live

    Actor: IAIEvaluationProvider

    A second AI pass checks the diff against acceptance criteria and ADRs, flags anything ignored or overbuilt, and drafts a suggested improvement to the prompt — kept separate from your rating, never blended into it.

    produces AIEvaluation
  8. 08

    Everything folds into one score

    live

    Actor: IScoringProvider

    Your rating, the AI's judgment, and the engineering metrics combine under a weighted formula you control into a single number attached to this PromptVersion — reproducible, and re-computable later if you change the weights.

    produces PromptScore
  9. 09

    It joins the pool every future task draws on

    live

    Actor: IRecommendationProvider (semantic + tag + historical blend)

    This execution, its score, and its tags become part of one shared database — not per-repo, not per-project, per machine. The next time you or a teammate starts something tagged or worded similarly, in this repo or a completely different one, this is what gets surfaced — even without exact tag overlap, since the embedding index catches what tag matching alone would miss.

    draws on Recommendation · query-time only Embedding
  10. 10

    It can promote itself — optional

    live

    Actor: AutoPromotionTrigger, or you via POST /prompts/…/activate

    By default a human still decides which PromptVersion counts as the official one — there's now a manual endpoint for that. Flip a policy toggle instead, and once a version's score clears a threshold — or beats the currently active version by a margin — the daemon activates it itself. No /promptops rate, no sign-off required.

    produces PromptVersion · Active PromotionPolicy

Next week, “implement feature Y” goes back through stage 01 — except now it gets classified and ranked against every scored PromptVersion on this machine, feature X's included, and if it scored well enough it may already be the active version without anyone having promoted it by hand.

Right now, this session?

No — it doesn't touch it.

Nothing in this pipeline sits between you and Claude mid-session. PromptOps observes and records; it never rewrites or augments what actually gets sent.

Next similar task?

Yes — which prompt you start with.

The version that scored well is what gets recommended next time. One that scored poorly gets surfaced with the evidence for why, not just quietly forgotten.

Every artifact this produces

One SQLite database, owned by a single local daemon.

ArtifactWhat it isStatus
PromptIdentity + name for a reusable promptlive
PromptVersionImmutable content, changelog, lineage back to the version it was edited fromlive
PromptMetadataTags, categories, owners — stored separately, queryable without touching contentlive

EngineeringMetricsBuild, test, coverage, static-analysis results for an executionlive
HumanEvaluationYour structured rating of an executionlive
AIEvaluationThe model's judgment of the same execution, kept separate from yourslive
PromptScoreWeighted combination of the above, attached to a PromptVersionlive
EmbeddingVector for a PromptVersion's name, tags, and content — powers semantic recommendation matchinglive
RecommendationRanked prompt versions for a new task — computed at query time, not storedlive
PromotionPolicyGlobal toggle + threshold/margin controlling whether scores can auto-activate a PromptVersionlive
Everything above lives in one SQLite database owned by a single local daemon that runs once per developer machine — nothing here is sent anywhere else.
